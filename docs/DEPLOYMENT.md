# Deployment Guide

WildScape Europe builds to static assets through Vite. The Release 1.0.0 deployment model is intentionally simple: validate the repository, build from `main`, publish `dist/`, and configure the optional public Mapbox token when authenticated maps are desired.[1] [2]

## Production build

```bash
npm install
npm run hygiene
```

The build output is written to `dist/`. Static assets generated by Vite can be cached aggressively, while `index.html` should be revalidated so new releases are visible after deployment.[2]

## Environment variables

| Variable | Required | Scope | Purpose |
|---|---:|---|---|
| `VITE_MAPBOX_ACCESS_TOKEN` | No | Public client variable | Enables authenticated Mapbox rendering when available. |

No private server secrets are required for Release 1.0.0 because application behavior is driven by local deterministic services. If a future backend is added, private secrets should live only in server-side infrastructure and should not use the `VITE_` prefix.

## Static host checklist

| Step | Action |
|---|---|
| Validate | Run `npm run hygiene` locally or in CI. |
| Build | Run `npm run build`. |
| Publish | Upload the contents of `dist/` to the static host. |
| SPA fallback | Route unknown paths to `index.html` when client routing is enabled. |
| Cache | Cache hashed assets for long durations and revalidate `index.html`. |
| Verify | Open the deployed app and test search, filters, map controls, dashboard, and booking simulation. |

## Release rollback

A rollback should redeploy the previous known-good `dist/` artifact or rebuild the previous validated commit. The maintainer should record the rollback reason in `CHANGELOG.md` and run `npm run hygiene` before cutting a follow-up fix.
