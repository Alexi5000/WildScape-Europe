# Deployment Guide

WildScape Europe builds to a static Vite artifact in `dist/`, which can be deployed to any static hosting platform that supports single-page application fallback routing.[1] The application is self-contained and can run without private credentials; a Mapbox token is optional for enhanced map rendering.[2]

> Deployments should be promoted only after `npm run validate` succeeds locally or in CI. That command runs type checking, linting, unit tests, and the production build.

## Pre-deployment checklist

| Check | Command or action |
|---|---|
| Install dependencies | `npm install` |
| Type safety | `npm run type-check` |
| Linting | `npm run lint` |
| Tests | `npm run test` |
| Production build | `npm run build` |
| Full gate | `npm run validate` |
| Local preview | `npm run preview` |

## Production build

```bash
npm install
npm run validate
npm run preview
```

The production output is written to `dist/`. Static assets generated by Vite can be cached aggressively, while `index.html` should be revalidated so SPA releases propagate correctly.[1]

## Environment variables

| Variable | Required | Production note |
|---|---:|---|
| `VITE_MAPBOX_ACCESS_TOKEN` | No | Add this public client token in the hosting platform only if authenticated Mapbox tiles are required. |

Only variables prefixed with `VITE_` are exposed to client code by Vite.[3] Do not place private server credentials in this frontend environment.

## Vercel

Vercel is a straightforward option for a Vite SPA because it detects the framework, runs the build command, and serves `dist/` with edge caching.[4]

| Setting | Value |
|---|---|
| Framework preset | Vite |
| Install command | `npm install` |
| Build command | `npm run build` |
| Output directory | `dist` |
| Optional environment | `VITE_MAPBOX_ACCESS_TOKEN` |

A minimal `vercel.json` can be used if explicit SPA fallback behavior is desired.

```json
{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "framework": "vite",
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}
```

## Netlify

Netlify can deploy the same artifact using a static publish directory and SPA fallback.[5]

```toml
[build]
  command = "npm run build"
  publish = "dist"

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

## GitHub Pages

GitHub Pages can host the generated `dist/` directory. If serving from a repository subpath, configure the Vite `base` option before building.[1]

| Deployment shape | Required consideration |
|---|---|
| Custom domain | `base: '/'` is usually correct. |
| Repository path | Set `base` to the repository path, such as `/WildScape-Europe/`. |
| SPA routing | Provide a fallback strategy or avoid deep direct links if the platform cannot rewrite to `index.html`. |

## Cache and header recommendations

| Path | Suggested cache policy |
|---|---|
| `/assets/*` | `public, max-age=31536000, immutable` |
| `/index.html` | `public, max-age=0, must-revalidate` |
| `/` | `public, max-age=0, must-revalidate` |

Security headers should be added in the hosting platform when supported.

| Header | Suggested value |
|---|---|
| `X-Content-Type-Options` | `nosniff` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | `camera=(), microphone=()` |
| `X-Frame-Options` | `DENY` |

## Post-deployment verification

After promotion, verify the product flows rather than only checking the homepage.

| Flow | Expected result |
|---|---|
| Landing page | Hero, effects, and primary calls to action render without runtime errors. |
| Campsite search | Query and filters return deterministic campsite results. |
| Map view | The map panel renders and controls respond. |
| Campsite details | Selection opens details, weather, and booking affordances. |
| Booking | Mock booking returns a confirmation and does not duplicate on retry. |
| Dashboard | User bookings, recommendations, and wishlist surfaces render. |
| Mobile viewport | Navigation and controls remain usable on narrow screens. |

## Rollback strategy

Static hosting platforms generally keep previous deployments. Prefer promoting the last known-good deployment in the platform dashboard. If rollback must be performed from Git, revert the faulty commit on `main` and allow the normal deployment pipeline to rebuild.

```bash
git revert <bad-commit-sha>
git push origin main
```

Avoid force-pushing `main` unless repository maintainers explicitly agree, because it rewrites shared history.
